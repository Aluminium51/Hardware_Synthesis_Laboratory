# 02 Clock Domains

## Why this document exists
This project is multi-clock by nature.
The camera interface, VGA output, and SCCB control do not all run in one clean shared domain.
Most serious integration bugs in this project will be clock-domain or timing-alignment bugs.

## Expected clock domains

### 1. Board/system clock domain
Name:
- `clk_100`

Source:
- Basys 3 onboard 100 MHz oscillator

Used for:
- top-level control
- SCCB controller timing generation
- clock generation input
- debug heartbeat if needed

## 2. VGA pixel clock domain
Name:
- `clk_vga`

Target use:
- 640x480 @ 60 Hz VGA timing generation
- framebuffer read address generation
- post-read filter stage
- VGA signal output timing

Used for:
- `vga_timing_640x480`
- `vga_reader_320x240`
- filter block
- RGB output register stage if present

TASK-001 bring-up note:
- The first hardware-passed VGA bring-up keeps logic in the `clk_100` domain and advances VGA timing with a 25 MHz `pixel_ce`.
- This avoids treating a fabric-divided flip-flop output as a project-wide clock.
- A true `clk_vga` from Clock Wizard / MMCM can still be introduced later if framebuffer integration or monitor behavior requires it.

## 3. Camera master clock output domain
Name:
- `cam_xclk`

Used for:
- clock driven out to the OV7670

Notes:
- this is generated by FPGA and sent to the camera
- it is not necessarily the domain used internally for capture logic
- the current top level normally drives a 25 MHz baseline XCLK, but the
  `sw[7]=1, sw[6]=1` full-VGA probe can switch that output to 50 MHz for a
  rate/noise experiment

## 4. Camera pixel clock domain
Name:
- `cam_pclk`

Source:
- returned by OV7670

Used for:
- pixel-byte sampling
- `HREF` / `VSYNC` interpretation
- RGB565 byte assembly
- framebuffer write enable and write address updates

This is the domain for the actual capture logic.

## 5. Line-buffer stream boundary
The new full-resolution streaming experiment still spans both `cam_pclk` and
`clk_100`, but it does not use the full-frame framebuffer as the boundary.

Ownership:
- camera domain writes each completed RGB565 pixel into the current line buffer
- camera domain emits a line-commit token when a scanline finishes
- VGA domain consumes committed lines one scanline later
- VGA domain owns the read address and read-enable for the active line

Cross-domain data:
- line ownership tokens
- write-pointer / read-pointer Gray-code synchronization
- line-ready status used to prevent underflow and overflow

This boundary is intentionally small and is the main point of risk in the
full-resolution experiment. Treat it as a separate timing problem from the
baseline framebuffer path.

## Domain ownership rules

### Camera domain owns
- camera byte sampling
- frame boundary detection from camera syncs
- write-side framebuffer control

### VGA domain owns
- output timing
- read-side framebuffer control
- scaling math
- filter selection and filter output

### System/config domain owns
- SCCB master
- camera init FSM
- debug state that does not require video-rate timing

## Main cross-domain boundary
The framebuffer is the intentional boundary between:
- camera write domain
- VGA read domain

This is the cleanest place to separate the two independent timing systems.

For the streaming experiment, the line ring replaces the framebuffer as the
active ownership boundary, but the same principle applies: one domain writes,
the other domain reads, and only narrow status/control signals cross between
them.

## Synchronization rules

### Allowed to cross through BRAM boundary
- pixel storage data
- write-side address and write-enable on camera port
- read-side address and read-data on VGA port

### Must be synchronized if exported as control/status
Examples:
- camera init done flag seen in VGA or top-level logic
- frame-ready or frame-count indicator used outside camera domain
- error flags crossing to another domain

Recommended methods:
- two-flop synchronizer for single-bit level signals
- toggle synchronizer for event pulses
- avoid unsafely sampling multi-bit counters across domains

## What must not be done
- do not sample camera pixel data in `clk_vga`
- do not generate VGA timing in `cam_pclk`
- do not move multi-bit camera buses into the VGA domain without a designed boundary
- do not assume camera and VGA are frequency-locked

## Reset strategy
Each major clock domain should get a synchronized reset.

Recommended resets:
- `rst_sys`
- `rst_vga`
- `rst_camcfg`
- camera-capture local reset behavior derived from frame sync where appropriate

Do not distribute a raw asynchronous pushbutton signal directly into every always block.

## Latency alignment requirement in VGA path
Even though BRAM is the main domain boundary, read latency still matters.

The VGA side must keep:
- `active_video`
- `hsync`
- `vsync`
- optional mode-valid flags

aligned with the returned pixel data.

If the BRAM read path takes one or more cycles, the control signals must be delayed by the same number of cycles.

## First-baseline simplification
For the first milestone:
- accept tearing from single buffering if necessary
- do not try to synchronize full-frame swaps
- keep all write logic in camera domain
- keep all read logic in VGA domain
- only synchronize small status flags to top-level debug outputs

## Later improvements after baseline
Possible future additions after baseline works:
- double buffering with controlled frame swap
- frame-ready handshake
- line-buffer-based filters
- more explicit capture/display ownership control
